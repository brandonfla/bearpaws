# Bearpaws v1.1.0 — QA pass + onboarding skill

**Released:** 2026-05-01

Hardening release on top of v1.0.0. Two rounds of independent static-analysis QA caught and fixed: a critical hook silent-fail bug, several content errors in the ADK and Cloud Run skills, broken `<see>`/`<include>` paths, and stale documentation. Adds a new always-first stack-agnostic skill (`bp:onboarding-to-a-project`) that lets the plugin handle stacks beyond the framework-specific domain skills.

## Headline changes

- **NEW skill — `bp:onboarding-to-a-project`.** Always fires first on any work that touches a codebase. Detects the stack from manifests (12+ ecosystems including JS/TS, Python, Go, Rust, Ruby, JVM, .NET, Elixir, Swift), reads `README.md`/`CLAUDE.md`/`AGENTS.md`, samples existing files, and runs the test command cold. Domain skills (`cloud-run`, `vite`, `google-adk`, etc.) layer ON TOP of the discovered project conventions, not as a replacement. Skipped for pure ideation prompts that don't touch code.
- **Hook hardening.** `hooks/session-start` now fails loudly (exit 1, stderr) if the bootstrap SKILL.md is missing/empty/unreadable instead of silently injecting a garbage payload. The matcher dropped `compact` to avoid 3× injection per session.
- **Bootstrap refresh.** New explicit lazy-load contract (`<see>` and `<include>` are advisory pointers, not directives). New 3-tier skill priority order (onboarding → process → domain) with an explicit ideation carve-out. Red Flags table merged from the deleted `_shared/` extraction file.
- **Schema validator** now scoped to `skills/*/SKILL.md` only; supporting files (HTML templates, external Anthropic docs, code examples) are intentionally out of scope.
- **Code-reviewer agent** got a Domain-Specific Review section in v1.0.0; carried forward.

## Content fixes (verified against current official docs)

### `bp:google-adk` + `bp:building-with-adk`
- Removed the fictional `@tool` decorator. Tools are plain Python functions auto-wrapped via `tools=[fn]`, or `FunctionTool(fn)` / `LongRunningFunctionTool(fn)` for explicit registration.
- Corrected callback names: `before_tool_callback`, `after_tool_callback`, `before_agent_callback`, `after_agent_callback`, `before_model_callback`, `after_model_callback` (was `before_tool_call` etc.).
- Corrected `ToolContext` parameter convention: must be named exactly `tool_context` with type `ToolContext` for ADK to inject session state.
- Updated import path: `from google.adk.tools import ToolContext` (no `tool` decorator import).
- Standardized on `from google.adk.agents import LlmAgent`.
- Updated model IDs: `gemini-flash-latest` (with stability caveat) and `gemini-2.5-pro` (verified canonical).
- Vertex AI Agent Engine deploy now shows the canonical `reasoning_engines.AdkApp(...)` wrap step before `vertexai.agent_engines.create(agent_engine=app, ...)`.
- Documented the `GOOGLE_GENAI_USE_VERTEXAI=TRUE` flag for switching from AI Studio to Vertex.

### `bp:cloud-run` + `bp:deploying-to-cloud-run`
- Replaced fictional `--cpu-always-allocated` flag with the real `--no-cpu-throttling`.
- Corrected `gcloud beta run services logs tail $SERVICE` to use `--project=$PROJECT_ID` (was `--region=$REGION`, which isn't the documented discriminator).
- Added a `gcloud logging read 'resource.type=cloud_run_revision'` fallback for environments without the beta logs-tail command.

### Skill descriptions
- All 10 domain-skill descriptions disambiguated: reference skills now read "Use as a reference for X mechanics — *what and why*", workflow skills read "Use when actively doing Y — hands-on workflow". Eliminates the routing collision between paired reference/workflow skills.

## Cleanup

- Deleted 6 orphan files: `skills/systematic-debugging/{CREATION-LOG,test-pressure-1..3,test-academic}.md`, `skills/writing-skills/render-graphs.js`.
- Deleted `skills/_shared/` directory; its single file (`red-flags-skill-discipline.md`, 10 lines) violated the extraction rule (>25 lines AND ≥2 consumers required). Content inlined back into the bootstrap.
- Refreshed `CLAUDE.md` to reflect Phases 1–3 ship state (XML schema is complete, hook silent-fail is fixed, matcher is `startup|clear`).
- Cleaned up stale `_shared/` references in `tests/schema-validator/README.md`, `tests/token-measurement/README.md`, `tests/token-measurement/measure.sh`, `skills/writing-skills/SKILL.md`.

| Bootstrap `additionalContext` | 2,907 bytes | 4,295 bytes | +48% (lazy-load contract + 4-tier skill priority + 4 Red Flags rows) |
| Process skills SKILL.md (14 inc. bootstrap) | 53,477 bytes | ~53,800 bytes | +0.6% (brainstorming and TDD got minor cross-references) |

| Schema violations | 0 | 0 | clean |
| Triggering test prompts | 18 | 19 (added onboarding) | +1 |

vs. superpowers v5.0.7 baseline:
- Bootstrap: 5,292 → 4,295 bytes = **-19% bytes / -23% tokens** (tiktoken cl100k_base)
- 14 process skills: 108,393 → ~53,800 bytes = **-50% bytes / -53% tokens**

The bootstrap grew vs. v1.0.0 because the lazy-load contract and skill-priority order are paid every session — that cost is real, but the net per-session payload is still well below superpowers v5.0.7. The bootstrap remains the deliberate "small price for big behavioral correctness" tradeoff.

## QA passes performed

Two rounds of 5/4 parallel static-analysis subagents:

**Round 1** (post-Phase 3 work, pre-1.0.0):
- Hook & runtime audit (POSIX, idempotency, output contract, failure modes)
- Token-density audit (tiktoken)
- Domain skill content audit (10 skills, fact-checked vs. official docs)
- Cross-file consistency audit (frontmatter, refs, orphans, namespace)
- Output contract + lazy-load semantics

**Round 2** (post-fixes, pre-1.1.0):
- Onboarding skill quality + integration with the rest of the suite
- Verification that the round-1 ADK/Cloud Run fixes are themselves correct (vs. official docs)
- Bootstrap quality + lazy-load contract effectiveness
- Cross-file consistency re-check (after `_shared/` removal, skill rename, orphan deletion)

All findings in both rounds either applied or documented; release notes capture the residuals.

## Known limitations (carried from v1.0.0)

- **Live integration suite** (`tests/claude-code/run-skill-tests.sh --integration`) has not been run on this codebase. Static analysis only.
- **Skill-triggering test suite** (`tests/skill-triggering/run-all.sh`) requires real Claude API calls; not run in this release cycle.
- **Gemini CLI compatibility** smoke-tested at v0.1.0; not re-verified on the XML-schema bodies.
- **A/B regression** vs. superpowers on identical coding prompts has not been performed — the "same behavioral performance" claim is supported by structural compatibility, not head-to-head eval data.
